# 本地日志条件组合查询工具

一个纯前端、可离线运行的本地日志搜索小工具，目标是用尽量少的技术栈完成“大文件多条件 AND 查询”。项目不依赖后端服务、不依赖数据库、不依赖 Elasticsearch / Lucene / Solr / Electron / Spring Boot，也不需要构建步骤。

## 功能说明

- 支持本地选择 `.log` / `.txt` 文件
- 支持拖拽打开本地日志文件
- 支持动态新增、删除查询条件
- 查询语义固定为 AND：同一行必须同时满足全部条件
- 自动忽略空条件，并在搜索前做 `trim` 与去重
- 默认大小写不敏感，并预留区分大小写开关
- 支持切换文件编码：`UTF-8`、`GB18030 / GBK`
- 支持中文、英文、带符号字符串查询
- 搜索结果显示行号、原始日志文本、命中数、耗时、进度
- 命中条件按原始字符串高亮，支持多条件并处理重叠
- 搜索过程支持取消
- 搜索结果边搜边返回、边搜边显示
- 结果区采用虚拟滚动，只渲染可视区域附近少量结果

## 运行方式

### 方式一：直接运行

直接双击打开 `index.html` 即可使用。

当前实现把 Worker 代码内联成 Blob 创建，因此通常不需要本地静态服务器。

### 方式二：如果你的浏览器对本地文件权限更严格

在项目目录执行下面任意一个最简单命令后，浏览器访问对应地址：

```powershell
python -m http.server 8080
```

然后打开：

```text
http://127.0.0.1:8080
```

### 常见问题

如果搜索时出现：

```text
The requested file could not be read, typically due to permission problems that have occurred after a reference to a file was acquired.
```

请优先按下面顺序处理：

1. 不要直接从压缩包预览区、只读目录或网络盘里打开日志文件，先复制到本地普通目录
2. 优先使用本地静态服务器方式运行页面，而不是直接双击 `index.html`
3. 重新选择一次文件后再搜索，避免旧文件句柄失效

当前实现已经改为“主线程分块读取文件，Worker 只负责扫描和匹配”，就是为了尽量规避这类浏览器权限兼容问题。

如果出现“英文条件能搜到、中文条件搜不到”，请优先检查页面里的“文件编码”：

- UTF-8：适合大多数现代日志文件
- GB18030 / GBK：适合部分 Windows 本地环境导出的中文日志

## 设计思路

### 1. 主线程职责

- 文件选择与拖拽交互
- 条件输入管理
- 搜索状态展示
- 结果虚拟列表渲染
- 命中高亮渲染

### 2. Worker 职责

- 分块读取文件
- 使用 `TextDecoder(stream)` 做 UTF-8 流式解码
- 处理 chunk 末尾半行残片
- 逐行执行条件匹配
- 进度回传
- 增量回传命中结果

### 3. 匹配策略

实现里明确拆分了以下核心函数：

- `tokenize(text)`
- `normalizeCondition(text)`
- `matchLineRaw(line, conditions, caseSensitive)`
- `highlightLine(line, conditions, caseSensitive)`

其中：

- `tokenize` 只用于生成辅助 hint token，做候选过滤
- `matchLineRaw` 才是最终命中判定，且永远基于原始整行文本

这意味着即使 `.`、`[`、`]`、`|` 等符号参与分词，最终匹配语义仍不变。

例如用户输入：

- `绑定队列`
- `etetet.topic`

只要原始行文本中同时包含这两个原始条件，就一定会命中。

## 大文件性能策略

这是本项目最核心的设计点。

### 1. 不整文件读入内存

搜索时使用：

- `File.slice(...)`
- `ArrayBuffer`
- `TextDecoder('utf-8')`

按 chunk 顺序扫描，不会一次性把整个文件完整读入内存。

### 2. 不在主线程做重活

逐行扫描、条件匹配、进度计算都在 Web Worker 中执行，主线程不会因为大文件搜索而假死。

### 3. 支持增量返回

Worker 会把命中结果按批次回传，界面可以边搜边显示，不需要等整文件扫描结束。

### 4. 只渲染窗口附近结果

结果列表使用固定行高的虚拟滚动，只渲染当前可视区域附近少量 DOM 节点，避免命中结果很多时页面卡顿。

### 5. 低内存思路

实现中不缓存整文件全文，只保存命中的结果行：

- 行号
- 原始日志文本

这对“大文件但命中相对少”的场景非常友好。

## 分词与最终匹配语义说明

项目内的分词分隔符覆盖了常见日志分隔内容，例如：

- 空格
- 制表符
- `[ ]`
- `( )`
- `{ }`
- `,`
- `.`
- `:`
- `;`
- `|`
- `/`
- `\`
- `-`
- `_`
- `=`
- 单引号
- 双引号
- 常见符号

但必须强调：

**分词只用于辅助过滤，不改变最终命中逻辑。**

最终判定严格基于原始整行文本执行 AND 查询。

所以：

- `etetet.topic` 会按完整原始字符串匹配
- 不会因为 `.` 被分词拆开而失真
- 中文短语也会按原始字符串正确匹配

## 高亮策略说明

高亮逻辑遵循以下规则：

- 基于原始字符串，不基于 token
- 先按原始行内容计算命中区间
- 多条件时采用“最长匹配优先”
- 对重叠范围做去重，避免 `mark` 标签嵌套错乱
- 输出 HTML 前会先对原始文本片段做转义，避免破坏日志原文和页面结构

## 手工测试步骤

### 1. 验证 AND 命中

1. 打开 `index.html`
2. 选择 `samples/sample.log`
3. 在两个条件框中分别输入：
   - `绑定队列`
   - `etetet.topic`
4. 点击“开始搜索”
5. 结果应至少命中 1 条，并能看到示例日志这一行

### 2. 验证删除条件后的结果变化

1. 在上一步结果基础上删除 `etetet.topic`
2. 保留 `绑定队列`
3. 重新搜索
4. 结果数应该增多，因为只剩单条件匹配

### 3. 验证高亮

1. 重新输入：
   - `绑定队列`
   - `etetet.topic`
2. 搜索结果中两段文本都应高亮
3. 高亮应直接作用于原始日志文本，而不是拆词后的文本

### 4. 验证大文件搜索时界面不假死

1. 准备较大的日志文件
2. 选择文件后开始搜索
3. 搜索过程中尝试滚动页面、点击条件输入框
4. 页面应保持可操作，进度、耗时、命中数持续更新

### 5. 验证取消搜索

1. 搜索大文件
2. 在搜索进行中点击“取消搜索”
3. 搜索应立即停止
4. 当前已返回的命中结果应保留

## 关键实现说明

### 1. 分块读取

Worker 中使用 `file.slice(offset, nextOffset).arrayBuffer()` 逐块读取文件，避免整文件进入内存。

### 2. UTF-8 流式解码

使用：

```js
decoder.decode(new Uint8Array(buffer), { stream: true })
```

这样 chunk 边界处即使切到中文字符中间，也不会乱码。

### 3. 半行残片处理

每次 chunk 解码后，把上一块遗留的 `remainder` 拼到当前文本前面，再统一切分行；最后一个不完整片段继续保留到下一块处理。

### 4. Worker 通信

Worker 通过三类消息和主线程通信：

- `progress`：进度、行数、命中数、耗时
- `results`：增量命中结果批次
- `done`：最终完成

### 5. AND 查询逻辑

先做条件标准化与去重，再做候选过滤，最终必须走 `matchLineRaw` 对原始整行执行全部条件 `includes` 判断。

### 6. 高亮逻辑

先找出全部命中区间，再按最长匹配优先去重，最后对每段文本分别做 HTML 转义后再拼装 `<mark>`。

### 7. 虚拟滚动逻辑

结果区使用固定行高：

- 用一个 spacer 撑起总高度
- 根据滚动位置计算当前可视区起止索引
- 只渲染可视区附近的少量结果

### 8. 性能设计

- 不读全文件
- 不建全量倒排索引
- 不把全部日志行渲染到 DOM
- 不把搜索重活放在主线程

## 已知限制

- 当前默认按 UTF-8 解码，未做 GBK / GB18030 自动识别
- 结果列表为了更稳定地做虚拟滚动，采用固定行高单行展示，超长日志可横向滚动查看
- 如果命中结果本身非常多，浏览器仍需要保存这些命中行文本，内存占用会随命中数上升
- 当前为纯字符串包含匹配，未加入正则、时间范围、字段级结构化过滤

## 为什么这个实现适合大文件且足够轻量

因为它抓住了大文件日志搜索里最关键的三个问题：

1. **读文件不能一次读完**
   采用 chunked reading + `TextDecoder(stream)`，保证中文和超大文件都能稳定处理。

2. **主线程不能做重活**
   搜索全过程在 Worker 中执行，页面负责响应用户操作和渲染，避免搜索时界面卡死。

3. **结果不能一次性塞进 DOM**
   采用虚拟滚动，只绘制当前可视区域附近的结果节点，即使命中很多条也不会因为 DOM 数量爆炸而卡顿。

因此这个方案既保留了纯前端、零后端、零数据库、零构建的轻量特性，又在大文件场景下具备比较可靠的稳定性和可维护性。
